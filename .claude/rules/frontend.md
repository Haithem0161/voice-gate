# Frontend Rules (React 19 + TanStack Router/Query + Orval + Tailwind CSS v4)

## Project Structure
```
frontend/
├── src/
│   ├── main.tsx                  # App entry: QueryClient, RouterProvider, providers
│   ├── app.css                   # Tailwind v4 entry: @import "tailwindcss"
│   ├── routes/
│   │   ├── __root.tsx            # Root layout (nav, <Outlet />)
│   │   ├── index.tsx             # Home page (/)
│   │   └── <route>.tsx           # File-based routes
│   ├── api/
│   │   ├── axios-instance.ts     # Custom axios instance with auth interceptor
│   │   ├── generated/            # Orval-generated hooks (DO NOT EDIT)
│   │   └── models/               # Orval-generated types (DO NOT EDIT)
│   └── components/               # Reusable UI components
├── orval.config.ts               # Orval codegen config
├── vite.config.ts                # Vite + TanStack Router plugin + Tailwind v4 plugin
└── package.json
```

## Orval Generated Code (CRITICAL)
- Files in `src/api/generated/` and `src/api/models/` are AUTO-GENERATED -- **NEVER edit them manually**
- To regenerate: start the backend, then run `make generate-api` (or `cd frontend && pnpm generate-api`)
- Orval reads the OpenAPI spec from `http://localhost:3000/api-docs/openapi.json`
- Generated hooks use TanStack Query (`useQuery`, `useMutation`) with the custom axios instance
- The `orval.config.ts` uses `mode: "tags-split"` -- each OpenAPI tag gets its own file
- The custom mutator at `src/api/axios-instance.ts` is referenced as `customInstance`

### When to Regenerate API Types
- After adding/modifying any `#[utoipa::path]` annotation on the backend
- After adding/modifying any `#[derive(ToSchema)]` on backend models
- After adding new paths or schemas to `#[openapi(...)]` in `main.rs`
- ALWAYS verify backend compiles and is running before regenerating

### Using Generated Hooks
```tsx
// Orval generates hooks like useListUsers, useCreateUser, etc.
import { useListUsers, useCreateUser } from '../api/generated/users';

function UserList() {
  const { data: users, isLoading, isError } = useListUsers();
  const createUser = useCreateUser();

  if (isLoading) return <div>Loading...</div>;
  if (isError) return <div>Error loading users</div>;

  return (
    <ul>
      {users?.map(user => <li key={user.id}>{user.name}</li>)}
    </ul>
  );
}
```

## TanStack Router Rules (File-Based Routing)
- Route files live in `src/routes/` -- the directory structure defines URL paths
- `routeTree.gen.ts` is auto-generated by the Vite plugin -- **NEVER edit it**
- The TanStack Router Vite plugin MUST come before the React plugin in `vite.config.ts`:
  ```ts
  plugins: [
    tanstackRouter({ target: 'react', autoCodeSplitting: true }),
    react(),
    tailwindcss(),
  ]
  ```

### Route File Pattern
```tsx
import { createFileRoute } from '@tanstack/react-router'

export const Route = createFileRoute('/users')({
  component: UsersPage,
  // Optional: loader for data fetching before render
  loader: async () => {
    // pre-fetch data
  },
  pendingComponent: () => <div>Loading...</div>,
  errorComponent: ({ error }) => <div>Error: {error.message}</div>,
})

function UsersPage() {
  const data = Route.useLoaderData()  // access loader data
  return <div>...</div>
}
```

### Root Layout (`__root.tsx`)
```tsx
import { createRootRoute, Outlet, Link } from '@tanstack/react-router'

export const Route = createRootRoute({
  component: () => (
    <div>
      <nav>
        <Link to="/" activeProps={{ className: 'active' }}>Home</Link>
        <Link to="/users" activeProps={{ className: 'active' }}>Users</Link>
      </nav>
      <Outlet />
    </div>
  ),
})
```

### Key Router APIs
- `<Link to="/path">` for navigation -- NEVER use `<a href>` for internal links
- `<Link to="/users/$userId" params={{ userId: '123' }}>` for dynamic params
- `Route.useParams()` for type-safe path parameters
- `Route.useSearch()` for type-safe query string parameters
- `<Outlet />` renders child routes inside layouts
- Dynamic routes: `users.$userId.tsx` maps to `/users/:userId`

## TanStack Query Rules (v5)
- Use Orval-generated hooks for API calls whenever available
- For manual queries: always provide a unique `queryKey` array
- `staleTime: 0` by default -- data refetches on mount if stale
- `gcTime: 5 * 60 * 1000` (5 min) by default -- unused cache is garbage collected
- Invalidate related queries after mutations:
  ```tsx
  const queryClient = useQueryClient()
  const mutation = useMutation({
    mutationFn: createUser,
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ['users'] })
    },
  })
  ```
- Always handle loading and error states from queries:
  ```tsx
  const { data, isLoading, isError, error } = useQuery({...})
  if (isLoading) return <Loading />
  if (isError) return <Error message={error.message} />
  ```
- Use `enabled` option for dependent queries (fetch B only after A succeeds)
- Retry defaults: 3 retries on client, 0 on server, with exponential backoff

## Form Handling Pattern (Zod + React Hook Form)
```tsx
import { useForm } from 'react-hook-form'
import { zodResolver } from '@hookform/resolvers/zod'
import { z } from 'zod'

// 1. Define Zod schema for validation
const createUserSchema = z.object({
  name: z.string().min(1, 'Name is required'),
  email: z.string().email('Invalid email'),
})
type CreateUserForm = z.infer<typeof createUserSchema>

// 2. Wire up react-hook-form with Zod
function CreateUserForm() {
  const { register, handleSubmit, formState: { errors } } = useForm<CreateUserForm>({
    resolver: zodResolver(createUserSchema),
  })

  // 3. Submit via Orval-generated mutation
  const createUser = useCreateUser()

  const onSubmit = (data: CreateUserForm) => {
    createUser.mutate(data, {
      onSuccess: () => { /* redirect or show success */ },
    })
  }

  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      <input {...register('name')} />
      {errors.name && <span>{errors.name.message}</span>}
      <input {...register('email')} />
      {errors.email && <span>{errors.email.message}</span>}
      <button type="submit" disabled={createUser.isPending}>Create</button>
    </form>
  )
}
```

## Tailwind CSS v4 Rules
- v4 uses the Vite plugin (`@tailwindcss/vite`) -- **NO `tailwind.config.js` or `tailwind.config.ts` file**
- Import in `app.css` using `@import "tailwindcss";`
- Customize the theme in CSS using `@theme` directive, not a JS config:
  ```css
  @import "tailwindcss";

  @theme {
    --color-primary: oklch(0.6 0.2 260);
    --font-sans: "Inter", sans-serif;
    --breakpoint-3xl: 120rem;
  }
  ```
- Custom theme values generate utility classes automatically (e.g., `bg-primary`, `font-sans`)
- Use `@theme { --color-*: initial; }` to override entire namespaces
- All styling via utility classes -- no custom CSS files unless absolutely necessary
- Responsive: `sm:`, `md:`, `lg:`, `xl:`, `2xl:`
- State: `hover:`, `focus:`, `disabled:`, `active:`, `group-hover:`
- Dark mode: `dark:` variant

## Component Conventions
- Functional components only (React 19)
- Component files: PascalCase (`UserCard.tsx`)
- Route files: match URL path pattern (`users.tsx`, `users.$userId.tsx`)
- Prefer composition over prop drilling -- use TanStack Query for server state
- Always handle `isLoading`, `isError`, and empty states from queries
